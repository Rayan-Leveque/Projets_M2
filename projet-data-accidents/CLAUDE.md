## Project context

M2 MIMO Université Paris 1 — *Projet Data en Python : Accidents de la route* (2025-2026). Coursework, not a software product. The deliverable is a PDF report (`report/`) plus the Python scripts and their outputs, due 2026-06-08, defended 2026-06-11.

The pedagogical workflow: **Python code is generated by an IA générative one question at a time, then analyzed and verified by the student.** The student has chosen **Claude Code** as the tool — declared in the dossier header per spec §1.4.1.

Because Claude Code is an *agent* (not a one-shot LLM), the team has decided to **drop the "prompt utilisé" section** from each dossier entry. Each question entry instead has:

1. Rappel de la question
2. **Travail avec l'agent** — contexte donné à l'agent + itérations & décisions (replaces the prompt + iterations sections of the spec)
3. Fichier Python : `python/questionXY.py`
4. Résultat / fichier généré : `results/reponseXY.*`
5. Vérifications réalisées + analyse + commentaires

This is a deliberate trade-off: the soutenance (20% of the grade) traditionally asks for a "stratégie de prompts efficaces" — the student will pivot the exposé toward "stratégie d'usage d'un agent IA conversationnel" instead. Be ready to support that angle when helping prepare the soutenance.

Code-writing posture: **write the code when asked**, explain the choices clearly enough that the student can defend the code at the soutenance. Don't refuse on pedagogical grounds.

## Coding style & priorities

When writing Python for this project, the code should be:

- **Transparente** — écrite en français (noms de variables, commentaires, messages) pour maximiser la lisibilité pédagogique.
- **Simple** — privilégier la clarté algorithmique à la robustesse. Un script qui lit un CSV ligne par ligne avec `csv.reader` est préférable à une abstraction pandas sophistiquée si cela rend le traitement plus lisible.
- **Fonctionnelle** — le code doit produire le bon résultat, mais une gestion d'erreur minimale suffit. Pas besoin de try/except sur chaque ligne si cela alourdit la lecture.
- **Explicite** — éviter les one-liners complexes, les compréhensions imbriquées, ou les générateurs invisibles. Privilégier les boucles explicites et les prints de progression.
- **Robustesse < lisibilité** — il vaut mieux un script simple qui fonctionne sur les données propres du projet qu'un script défensif de 300 lignes avec 15 cas d'erreur.

The student is **doing all coding solo** and treats **facultative questions as in-scope** (Q2.3, Q2.4, Q5.3, Q6.5).

## Architecture

The repository structure is **prescribed by the project spec** (`docs/Projet Data en Python - Accidents de la route 26 04 2026.md`, §1.4.1) and must be preserved:

- `data/raw/` — raw BAAC files downloaded from data.gouv.fr (caracteristiques, lieux, vehicules, usagers).
- `data/utf8/` — UTF-8 re-encoded files with `,` separator (output of Q1.1).
- `data/enriched/` — files where enumerated codes are replaced with their labels (output of Q1.3). All Q2+ work reads from here.
- `python/` — one script per question, named `questionXY.py` (e.g. `question22.py` for question 2.2). Naming is graded.
- `python/common.py` — shared paths (`DATA`, `RAW`, `UTF8`, `ENRICHED`, `RESULTS`) and the year range constants. Scripts should import from here rather than hard-coding paths.
- `results/` — execution traces and generated files, named `reponseXY.*` (matching naming).
- `report/dossier.md` — the report itself, written in Markdown at the repo root of `report/`. Edit this file directly as questions are completed.
- `report/dossier.tex` — the LaTeX version of the report, **a committed artifact** generated from `dossier.md` by `report/build_tex.py`. Targets **pdflatex** (Unicode handled via `\DeclareUnicodeCharacter`, front-matter folded into a title page + auto TOC).
- `report/build_tex.py` — one-shot Markdown→LaTeX converter (stdlib only). Run `python build_tex.py` from `report/` after editing `dossier.md` to regenerate `dossier.tex`. It is **not** wired into the Makefile.
- `report/Makefile` — the LaTeX builder. `make` compiles `dossier.tex → dossier.pdf` (pdflatex, double pass for the TOC); `make clean` removes aux files. It only compiles the `.tex`; it does **not** regenerate it from the `.md`. Build prereqs: `texlive-latex-base texlive-latex-recommended texlive-fonts-recommended texlive-lang-french`.

### Data pipeline

Q1.1 → Q1.3 produce intermediate datasets that downstream questions consume:

```
raw/  ──Q1.1──▶  utf8/  ──Q1.3──▶  enriched/  ──▶  Q2.x, Q3.x, Q4.x, Q5.x, Q6.x
```

Q2.x checks referential integrity via `NumAcc` (joins all four files) and `Num_Veh` (joins vehicules ↔ usagers).

### Group-specific year range

Three student groups, each analysing a different 11-year window:
- Group 1 → 2005–2015
- Group 2 → 2006–2016
- Group 3 → 2007–2017 (also must do all *facultative* questions)

`python/common.py` currently sets `ANNEE_DEBUT=2005, ANNEE_FIN=2017` as a placeholder — confirm the user's group before running anything that depends on the range.

### Format quirks (from the spec)

- BAAC vehicle files come with both `-` and `_` in their names; the `vehicules-immatricules-baac…` files are *not* the right ones.
- Missing values appear as empty cell, `.`, `-1`, or sometimes `0` depending on the attribute. Q4 depends on distinguishing these from real numeric values.
- Spec restricts analysis to mainland France (exclude DOM départements).
